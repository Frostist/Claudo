Architecture Approaches

How should Godot, the AI agents, and the Games Master connect?

A

Pure Godot — HTTP calls direct from GDScript

Godot calls Gemini & Claude APIs directly using HTTPRequest nodes. Each NPC stores its memory as a GDScript Dictionary. The GM is a background Godot thread polling game state.

Pros

  • One codebase, no extra services
  • Easy to ship / no server to run

Cons

  • GDScript is painful for complex agent logic
  • No clean MCP interface for the GM
  • NPC-to-NPC comms are messy to coordinate
B

Python Backend — FastAPI + WebSocket bridge ⭐ Recommended

Godot handles rendering & input only. A local Python FastAPI server runs all agent logic — Gemini for NPCs, Claude Opus for the GM. Godot and Python talk over WebSocket for real-time updates. NPC memory = JSON files on disk. The GM reads everything via an MCP-style tool interface built into the server.

Pros

  • Python has first-class Anthropic & Gemini SDKs
  • Clean separation: Godot = game, Python = brains
  • MCP server is trivial to build here
  • NPC-to-NPC comms are just function calls
  • Easy to debug agent logic independently

Cons

  • Player needs Python running locally alongside Godot
  • Two things to launch (manageable with a shell script)
C

Node.js Bridge — TypeScript agent server

Same split as B but the backend is TypeScript/Node.js instead of Python. Godot ↔ Node.js over WebSocket. Uses the Anthropic TS SDK and Google Gemini JS SDK. Better if you prefer JS/TS over Python.

Pros

  • TypeScript is strongly typed, great for agent state
  • Single JS runtime, no Python dependency
  • Anthropic & Gemini both have great JS SDKs

Cons

  • Same two-process issue as B
  • Slightly less mature AI tooling ecosystem vs Python

How it flows (Option B/C)

Godot (render + input)
  ↕ WebSocket
Agent Server (Python or TS)
  ├── NPC Agents → Gemini Flash/Pro
  ├── Games Master → Claude Opus 4.7
  ├── Memory Store (JSON files per NPC)
  └── MCP tools (read chat logs, push game events)
